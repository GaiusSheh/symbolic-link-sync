# SymLiSync

Windows Junction 管理工具，让网盘目录之间的符号链接在多台设备上自动维护。

## 为什么需要SymLiSync？

OneDrive、Google Drive 等网盘**不会同步 Windows Junction**（目录符号链接）。Junction 是一种让一个目录"出现在"另一个位置的机制——比如把代码仓库链接进研究资料目录，或把分散的数据目录整合到一个入口。

问题在于：Junction 里存的是本机绝对路径，换一台设备路径就变了，同步过去也是坏的。SymLiSync 用模板路径（`{onedrive}/X`）代替绝对路径，在每台设备上按本机实际路径重建 Junction，让"目录快捷方式"真正跟着配置文件走。

## 我不使用网盘跨设备同步，可以从 SymLiSync 获得什么？

单台设备上 Junction 也会因为目录被移动、重命名而失效，手动重建很繁琐。SymLiSync 可以：

- **统一管理**所有 Junction，状态一览无余
- **自动修复**：移动目录后程序检测到变化，自动更新路径并重建
- **扫描导入**现有 Junction，无需从头配置
- **开机自启**，后台静默运行，有问题才通知你

## SymLiSync 如何工作？

程序常驻系统托盘，后台监听文件系统事件（目录创建、移动、重命名）和关键配置文件 `symlinks.json` 的变化。检测到链接失效或目录变动时，自动更新配置并重建 Junction；也按设定的时间间隔定期全量检查。所有操作都在本机完成，不依赖任何网络服务。

## 我是否需要注册账号？

不需要。SymLiSync 是纯本地程序，无账号、无联网、无遥测。

## 我的数据如何储存？

所有链接配置存在一个 `symlinks.json` 文件里，位置由你在首次运行时选择，是标准 JSON，可以直接查看和编辑。将它放在网盘目录后，其他设备上的 SymLiSync 会在同步到更新时自动按新配置建立或修复 Junction，无需任何手动干预，所有设备的目录结构保持一致。程序设置（`settings.json`）和运行状态（`state.json`）存在程序同目录的 `data/` 文件夹下，不随 `symlinks.json` 同步到其他设备。

---

## 功能一览

- **自动建链**：配置文件里定义的 Junction 在每台设备上自动创建，无需手动 `mklink`
- **实时监控**：程序常驻系统托盘，实时检测链接失效并自动修复
- **移动自愈**：目录被移动或重命名后，程序自动更新配置并重建 Junction
- **新建自托管**：在受监听目录内创建的新 Junction 自动纳入管理
- **扫描导入**：扫描已有 Junction 并批量导入管理
- **多设备同步**：配置文件可放在网盘目录，多台设备共享同一份链接定义，以 `{onedrive}`、`{gdrive}` 等占位符代替各台机器上不同的根目录路径

## 安装

运行安装器 `SymLiSync-Setup-x.y.z.exe`（中文向导，无需管理员权限，可自定义安装目录）。安装会放入两个程序并：

- **`SymLiSync-Tray.exe`**：托盘主程序（GUI）
- **`symlisync.exe`**：命令行工具，安装时自动加入 PATH（供脚本/AI 代理调用，见「命令行」一节）
- 可选勾选：**开机自启**、**加入 PATH**
- 默认启用资源管理器右键菜单与快捷键（可在设置中关闭）

卸载在「设置 → 应用」或开始菜单完成，会自动清理右键菜单、自启、PATH，并询问是否保留你的数据（`data/`）。

> 从源码构建：`build.ps1` 产出两个 exe 到 `dist/`；`installer/SymLiSync.iss` 用 Inno Setup 编译成安装器。

## 初始配置

### 第一步：选择配置文件位置

首次运行程序时会弹出向导。

**全新使用**：选择「创建新配置」，浏览到一个**网盘目录**（如 OneDrive、Google Drive），程序在该目录下新建 `symlinks.json`。

**已在其他设备使用过**：选择「使用已有配置」，找到网盘里已有的 `symlinks.json` 文件。

> 跳过向导会使用程序目录下的 `data/symlinks.json`，跨设备同步功能不可用。

> **已有配置文件**：选「创建新配置」但所选目录下已存在 `symlinks.json` 时，程序会询问是继承现有配置还是清空重建。

### 第二步：注册 Base 路径

Base 是对本机网盘根目录的命名，例如：

| Base 名 | 本机路径 |
|---------|---------|
| `onedrive` | `C:\Users\Alice\OneDrive` |
| `gdrive` | `G:\My Drive` |

不同设备上的 Base 名相同，路径各自填写本机的真实位置。全局链接里用 `{onedrive}/X` 这样的模板路径，程序按本机 Base 展开为实际路径。

若当前机器尚未注册，向导结束后会自动进入注册窗口。程序会自动检测已安装的 OneDrive、Google Drive、Dropbox、iCloud 并预填路径，按需调整即可。之后可在「管理同步目录」或「设置」中修改。

### 第三步

点击「扫描链接」，选择网盘根目录扫描。程序会发现网盘中已有的 Junction 并自动纳入管理——绝大部分链接无需手动配置。

## 日常使用

### 新建链接

点击状态窗口「新建」，填写以下字段：

- **编号**：唯一标识符（建议英文加短横线，如 `my-data-link`）
- **描述**：可选说明
- **符号链接位置**：默认填「父目录 + 名称」两栏，程序在父目录下新建一个名为该名称的符号链接（零误删）。勾选「替换以下目录」后，名称栏灰掉，所选目录**本身**被替换为符号链接（若该目录非空，会弹窗确认后删除其内容）。
- **符号链接指向**：符号链接指向的真实目录（如 `{onedrive}/files/data`）

保存后程序自动建立 Junction。

### 编辑与删除

双击任意行打开编辑对话框。右键菜单支持「删除管理」（仅移除管理记录，保留符号链接）或「删除管理和符号链接」（删链接本身，**不影响其指向的真实数据**）。Shift / Ctrl 多选后可批量删除，Delete 键仅删记录（均有确认弹窗）。

### 扫描已有 Junction

「扫描链接」扫描指定目录下所有未被管理的 Junction，结果实时显示。逐条「导入管理」或「不管理」（忽略）。

**Rebase**：扫描完成后，若所选目录是某个已注册 Base 的**父目录**，程序会询问是否 Rebase。例如 `{onedrive} = C:\Users\Alice\OneDrive`，扫描 `C:\Users\Alice` 时可将其注册为新 Base `{home}`，所有 `{onedrive}/X` 自动替换为 `{home}/OneDrive/X`，便于将来 OneDrive 整体迁移时只改一处路径。若所选目录与已有 Base 完全无关，程序也会询问是否将其注册为新 Base。

> **注意**：Rebase 只更新当前设备的路径模板，变更会随 `symlinks.json` 同步到其他设备，但其他设备需在「管理同步目录」中手动为新 Base 填写本机路径，否则相关链接会变为离线或缺失状态。

## 资源管理器集成（右键菜单 + 快捷键）

默认开启，可在设置中关闭。

### 右键菜单

Win11 在「显示更多选项」内有 **SymLiSync** 级联菜单（经典菜单，写入 HKCU，无需管理员）：

- 右键一个**文件夹**：新建指向此文件夹的符号链接 / 将此文件夹替换为符号链接 / 原地创建符号链接
- 文件夹**空白处**右键：在此处新建符号链接 / **粘贴为符号链接**

「粘贴为符号链接」：先在资源管理器里 `Ctrl+C` 复制一个文件夹，再到目标目录空白处选它，即在当前目录建一个指向被复制文件夹的符号链接并纳管。

### 快捷键（仅资源管理器内生效）

- **Ctrl+Q**：粘贴为符号链接到当前文件夹（配合 `Ctrl+C` 复制文件夹）
- **Ctrl+J**：对选中的文件夹**原地创建**符号链接（名为「X - 符号链接」），之后可 `Ctrl+X` 移走、改名，全程被监听自动跟踪

> 快捷键经低级键盘钩子实现，**仅当资源管理器在前台时拦截**，在其他程序里 Ctrl+Q/Ctrl+J 照常透传，互不影响。

## 命令行（CLI）

安装后 `symlisync` 命令在任意终端可用（已加入 PATH）。直接操作核心，不经托盘；每条命令返回退出码（0 成功 / 非 0 失败），支持 `--json`（便于脚本与 AI 代理解析）。与 GUI 共享同一份 `data/settings.json`（即同一个 `symlinks.json`），也可用 `--db <路径>` 覆盖。

```
symlisync list   [--json]          # 列出受管条目 + 状态
symlisync status [--json]          # 按状态汇总计数
symlisync add <符号链接位置> <符号链接指向> [--name 编号] [--force]
symlisync remove <编号> [--with-link]   # --with-link 同时删除符号链接本身
symlisync sync                     # 重建断链 / 待建链接
```

示例：`symlisync add "{onedrive}/Codes/proj/data" "{onedrive}/files/data" --name proj-data`（也可用绝对路径）。

> 适合让 AI 代理批量建链：一句话让它跑若干 `symlisync add` 即可，无需手动编辑 `symlinks.json`。

## 跨设备使用

**核心原则**：`symlinks.json` 放哪里，同步就靠哪里——程序不传输文件，由网盘自动同步。

### 新设备接入

1. 安装 SymLiSync，首次运行弹出向导
2. 选择「使用已有配置」，找到网盘里的 `symlinks.json`
3. 注册本机 Base 路径（Base 名和其他设备一致，路径填本机的）
4. 立即同步，所有链接自动在本机创建

### 未注册的 Base

若 `symlinks.json` 中的全局链接使用了当前设备未注册的 Base（如 `{gdrive}`），这些链接在状态窗口中会显示为 ⚠️ 缺失，程序无法为其创建 Junction。

处理方式：托盘右键 → 管理同步目录，为该 Base 名填写本机对应路径（如 `G:\My Drive`）。注册后程序自动解析并建立链接。

若该网盘在本机确实不存在（如某设备没有 Google Drive），可在「管理同步目录」中勾选「此机器不使用」——这些条目会从列表中隐藏，不影响其他链接正常工作。

### 注意事项

- 每台设备需独立注册 Base 路径，配置存在 `symlinks.json` 的 `machines` 字段里，会随文件同步到所有设备（各设备只读取自己那条，不会使用其他设备的路径）
- 若 Base 目录发生整体迁移（如 OneDrive 换了位置），可使用「扫描链接」中的 **Rebase** 功能批量更新路径模板（参见上方说明）

## 状态与自动修复

### 状态图标

| 图标 | 含义 |
|------|------|
| ✅ 正常 | Junction 存在且目标可达 |
| ➕ 待建 | 目标存在但 Junction 尚未创建 |
| ❌ 断链 | Junction 存在但目标不可达 |
| ⚠️ 缺失 | Junction 和目标均不存在（可能未同步） |
| ⚠️ 空目标 | Junction 正常但目标目录为空 |

### 新建 Junction 自动托管

在已注册 Base 目录内新建 Junction 时，程序实时检测并纳入管理（基于文件系统事件，通常 1 秒内）——编号无冲突时静默注册（banner 通知），编号冲突时弹窗提示输入新编号（可跳过）。

### 自动修复场景

Windows 下断链通常由两种情况引起：

1. **目标目录被移动或重命名**——Junction 指向的真实目录不见了，Junction 变为断链（❌）
2. **Junction 目录被移动**——在 Windows 下"移动"一个 Junction 实际上是把目标目录的全部内容复制到新位置后删除原内容，Junction 本身不断开但目标变空（⚠️ 空目标）

本程序提供自动修复：

1. **目标目录被移动**：Junction 变为断链。程序收到文件系统事件后立即在最近 8 秒的创建记录中搜索新位置，找到则自动更新配置并重建 Junction，右下角显示通知；未找到则弹对话框，提供「立即修改」入口。

2. **Junction 本身被重命名**：程序在同一父目录下搜索指向相同目标的 Junction，找到后自动更新配置，静默完成。

3. **目标目录被整体移动（含重命名）**：watchdog 捕获目录移动事件后自动批量更新所有涉及该路径的条目，静默完成。

4. **Junction 被 Explorer 跨目录移动**：Windows 的 cut+paste 会把 Junction 目标内容复制到新位置。程序检测到后自动将内容移回、在新位置重建 Junction，发通知。

若无法自动修复，则需要手动干预。需要手动干预的情形：

**未被识别的变更**：程序未运行期间发生的移动/重命名，或目录移动到未被监听的范围（外部硬盘、未注册 Base 之外）。在状态窗口双击对应条目手动修改路径即可。

**空目标**：目标目录为空时显示黄色提示，不弹窗。若程序运行期间目标从有内容**变为空**，会弹窗提示（可能是 Junction 被移走）。处理方式：
- 打开编辑对话框点「确定」→ 确认当前为空的状态为正常
- 右键 → 「（重新）链接 / 确认」→ 同上
- 弹窗里点「稍后处理」→ 关闭提示，黄色状态保留
- 若知道内容去了哪里，编辑条目更新目标路径

### 本地和离线管理

**状态窗口分区**：列表上方是所有设备共享的全局链接。下方按分隔线分为四个区：

- **已忽略的全局配置**（默认折叠）：使用了本机标记为「此机器不使用」的 Base 的全局链接，仅供查阅。

- **本机管理**：链接路径或目标路径在所有已注册 Base 之外（如绝对路径 `D:\LocalData`），其他设备无从解析，因此只在本机维护，不同步给其他设备。若日后将该目录纳入某个 Base，链接会自动升级为全局链接，对所有设备可见。

- **离线配置**：其他设备上定义的本机链接，在当前设备上尚未配置。双击打开配置对话框，程序会尝试按本机 Base 自动预填路径，确认后创建链接；若路径落入已注册 Base 内，会创建为全局链接，否则创建为本机链接。

- **未托管的扫描结果**：扫描时发现但尚未导入管理的 Junction，可逐条导入或忽略。

## 设置

托盘右键 → 设置：

| 选项 | 说明 |
|------|------|
| 检测间隔 | 定期检查断链的时间间隔（分钟） |
| 开机自启 | 登录时自动启动（写入注册表 Run 键；安装时也可勾选） |
| 关闭时收起到托盘 | × 按钮隐藏到托盘而不退出 |
| 添加到资源管理器右键菜单 | 启用/关闭文件夹右键的 SymLiSync 菜单（默认开） |
| 启用快捷键 | 启用/关闭资源管理器内的 Ctrl+Q / Ctrl+J 快捷键（默认开） |
| 同步目录（Base） | 可在设置窗口内直接查看和修改本机的 Base 路径，包括重命名 |

退出程序请使用托盘右键菜单「退出」。

## symlinks.json 结构参考

配置文件是标准 JSON，可直接查看和编辑：

```json
{
  "machines": {
    "LAPTOP":  { "onedrive": "C:/Users/Alice/OneDrive" },
    "DESKTOP": { "onedrive": "D:/OneDrive" }
  },
  "symlinks": [
    {
      "id": "my-data-link",
      "target": "{onedrive}/files/data",
      "link": "{onedrive}/Codes/my-project/data-link",
      "target_override": { "DESKTOP": "D:/LocalData/data" }
    }
  ],
  "local_data": {
    "LAPTOP": { "symlinks": [...], "scanned": [...] }
  }
}
```

- **machines**：各设备的 Base 路径，随文件同步到所有设备（各设备只读自己那条）
- **symlinks**：全局链接（模板路径），所有设备共享
- **symlinks[].target_override**：按机器名覆盖 target 路径，用于某台设备上目标路径与全局模板不同的情形
- **local_data**：各设备的私有数据，仅由本机写入，其他设备不解析
